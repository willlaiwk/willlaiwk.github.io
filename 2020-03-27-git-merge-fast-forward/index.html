搞懂 Git Merge 的 fast-forward (快進模式)

March 27, 2020

...

不知道大家在 merge 時有沒有產生一個疑問，一樣都是下 git merge 指令，但有時候會要求你建立 commit 點，有時候又不會，到底是為什麼呢？ 答案就是 merge 的 fast-forward(快進模式)

什麼是 fast-forward 快進模式

使用 fast-forward(快進模式)時，Git 會在 merge 時檢查：「併進來的分支可不可以直接接續在當前的 commit 點」。 若可以則直接往後接；若不行則產生一個 merge commit 點，也就是原本 merge 的動作。

若想在 merge 時使用 fast-forward ，可以下指令：git merge --ff ...，

不過 Git 預設 merge 就是使用 fast-forward 的，所以下指令： git merge ... 就可以囉。

但有時你就是想在 merge 時產生 commit 點，那可以使用 git merge --no-ff 指令，強制不使用 fast-forward。

情境 1 - 可以使用 fast-forward 的情況

想像一下，我們為了開發系統的新功能，所以從 master 切出 feature 分支。

現在 feature 開發完了，並記錄了兩個 commit 點，現在我們想把他合併回 master。

1-1. 用 fast-forward

我們切回了 master 分支，這時我們會在 Add A.txt 的那個 commit 點上。

然後執行以下指令：

$ git merge feature --ff
or
$ git merge feature

1-2. 不用 fast-forward

我們一樣切回了 master 分支，這時我們會在 Add A.txt 的那個 commit 點上。 現在我們強制不使用 fast-forward，所以執行以下指令：

$ git merge feature --no-ff

這時會要求輸入 commit，這裡我簡單紀錄一下：
Merge branch 'feature' into master

這時你會發現，與剛上面的1-1範例不同。我們建立了一個 commit 點:「Merge branch 'feature'」，然後保留了原本 feature 分支的線圖狀態。 如下圖：

情境 2 - 無法使用 fast-forward

如 情境 1，我們同樣切 feature 分支開發新功能，但當我們開發完功能時發現， master 居然已經先被別人更新了。這時的線圖如下圖：

這時我們切回 master 應該會在 Add Z.txt 這個 commit 點上。

然後輸入以下指令：

$ git merge feature --ff
or 
$ git merge feature

當你輸入完你會發現，Git 要求你一定要輸入 commit 訊息，然後輸入完線圖就會變成下面這樣。如下圖：

會發現這裡無法使用 fast-forward 是因為， feature 是從 Add A.txt 這個 commit 點切出去的，照理說使用 fast-forward 的話，Add B.txt 與 Add C.txt 應該要接在 Add A.txt 後面，可是這時候卻多了個 Add Z.txt 的點，所以在沒辦法順利接回的情況下，就必須走正常的 merge 流程，也就是建立一個 merge commit 點。

若這個狀況下，你真的就想要使用 fast-forward…

你可以考慮先在 feature 分支 rebase master 分支，再使用 fast-forward。

不過 rebase 又是另一個故事了…

總結

  • Git Merge 預設就是使用 fast-forward。
  • 使用 fast-forward 的指令：git merge or git merge --ff
  • 強制不使用 fast-forward 的指令：git merge --no-ff
  • fast-forward 模式只有在: 併進來的分支可以直接接續在當前的 commit 點後面的情況下才會成功，不然一律都需要建立 merge 的 commit 點。
